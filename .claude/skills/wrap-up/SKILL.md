# /wrap-up — Save Session to Branch Shard

Save session progress to a per-branch session shard, write a branch-scoped handoff, capture to brain with worktree metadata. NO git operations. NO writes to PROGRESS.md (consolidation happens at merge to main via /push).

## Triggers
- `wrap up`
- `save progress`
- `done for now`
- `lets wrap`

## Workflow

### Step 0: Identity

Run in parallel:
```bash
WORKTREE_ROOT=$(git rev-parse --show-toplevel)
BRANCH=$(git branch --show-current)
COMMON_GIT_DIR=$(git rev-parse --git-common-dir)
CANONICAL_ROOT=$(cd "$(dirname "$COMMON_GIT_DIR")" && pwd)
CANONICAL_NAME=$(basename "$CANONICAL_ROOT")
SHARD_KEY=$(echo "${BRANCH:-detached-$(git rev-parse --short HEAD)}" | tr '/' '-')
```

`CANONICAL_NAME` is the project identity for brain operations. NEVER substitute `basename($WORKTREE_ROOT)` — inside a worktree those differ.

**Ensure `.gitignore` lets `.claude/sessions/` and `.claude/handoffs/` ride merges.** Handle the project's existing `.gitignore` shape carefully — do NOT blindly append:

1. **If `.gitignore` doesn't exist**, create it with exactly:
   ```
   .claude/*
   !.claude/sessions/
   !.claude/handoffs/
   ```

2. **If `.gitignore` exists**, do this decision tree:
   - Grep for `!.claude/sessions/` AND `!.claude/handoffs/`. If both negations are already present (any line, any form), **leave the file untouched** — set gitignore status to `unchanged`.
   - If `.claude/*` or `.claude/` is ignored but the negations are missing, append the two negations directly after the existing `.claude` ignore rule. Add a marker comment line `# Agent-View: track session shards + handoffs` above the negations so the additions are visible in diffs. Set status to `negations appended`.
   - If `.claude/` isn't ignored anywhere in the file, append the full three-line block at the END of `.gitignore` (the negations are no-ops but the block documents the convention for future edits). Set status to `negations appended (no-op)`.
   - **If `.gitignore` looks generated** — check for marker comments like `# AUTO-GENERATED`, `# Generated by`, `# DO NOT EDIT`, or if the file is a symlink — **do NOT modify it**. Set status to `skipped (generated)` and tell the user in Step 7 to add the negations to the source-of-truth ignore manually.

3. **Never `git add` `.gitignore`** from /wrap-up. Leave the modification unstaged so the user reviews and stages it on their next commit. The point is visibility, not silent commits.

### Step 1: Assess What Happened

Review the conversation to identify:
- What was worked on (features, fixes, refactors, decisions)
- Key decisions made and their rationale
- What's still remaining from the current plan/task
- Any blockers discovered

### Step 2: Append to Branch Shard

Path: `$WORKTREE_ROOT/.claude/sessions/$SHARD_KEY.md` (create directory if needed).

Append a session block:
```markdown
## Session — YYYY-MM-DD HH:MM (wt: <basename WORKTREE_ROOT>)

### Work Done
- ...

### Decisions
- ...

### Where Left Off
- <specific state — file paths, line numbers, what's partially done>
```

Multiple wrap-ups on the same branch append additional `## Session` blocks to the SAME shard. The shard is the branch's full narrative. PROGRESS.md is NOT touched.

### Step 3: Update TASKS.md with Claim Annotations

Check off completed tasks with branch annotation:
```markdown
- [x] Task description (done: $BRANCH @ YYYY-MM-DD HH:MM)
```

Mark in-progress tasks:
```markdown
- [~] Task description (in-progress: $BRANCH @ YYYY-MM-DD HH:MM)
```

The annotation makes parallel claims visible. Two branches checking off the same task will produce a real merge conflict on the annotation — that's the intended signal.

### Step 4: Brain Captures with Worktree Metadata

Use `capture_thought` for each meaningful occurrence. For every call, prepend a metadata line to `text`:
```
[branch:$BRANCH | wt:$(basename $WORKTREE_ROOT) | project:$CANONICAL_NAME]
<the actual capture text>
```

Set `project_ref: $CANONICAL_NAME`. Use `source: "claude-code"`. Do NOT pass `brain_id` — let smart-capture route.

Categories (one capture per category that occurred):
- **Decisions** — Architectural, technical, or strategic choices with rationale
- **Insights** — Patterns noticed, lessons learned, cross-project connections
- **People** — Anyone mentioned in meaningful context
- **Blockers & Resolutions** — Problems hit and how they were solved

### Step 5: Append to BRAIN.md

In `$WORKTREE_ROOT/BRAIN.md` (git-tracked, lives in the worktree, merges normally), append under `## Session Log`:

```markdown
### YYYY-MM-DD / $BRANCH
- [decision] — ...
- [insight] — ...
```

The branch is part of the heading so concurrent worktree entries on the same date don't collide at merge.

### Step 6: Write Branch-Scoped Handoff

Path: `$WORKTREE_ROOT/.claude/handoffs/$SHARD_KEY.md` (create directory if needed).

```markdown
# Session Handoff — $BRANCH
Generated: YYYY-MM-DD HH:MM
Worktree: $WORKTREE_ROOT

## What We Were Working On
<1-2 sentences describing the active task/feature>

## Remaining Work
<from TASKS.md — unchecked items relevant to this branch>
<from PLAN.md — unfinished steps, if PLAN.md exists>

## Key Decisions This Session
<1-2 liner summaries of decisions made, if any>

## Kickstart Prompt
> <ready-to-use prompt for the next session on THIS branch.
> Specific: file paths, function names, line numbers.
> Should be specific enough that a fresh Claude Code instance
> can continue without the user re-explaining anything.>
```

There is NO singular `.claude/handoff.md` anymore. If one exists from a previous version of this skill, delete it.

The kickstart prompt is still the most important part. Concrete and actionable — not "continue working on the feature" but "Implement the webhook handler in src/api/webhooks.ts — the route is scaffolded but the payload parsing and DB insert are still TODO."

### Step 6.5: FHE Artifact Auto-Push (fhe-hq Files)

If this session produced standalone **deliverable** artifacts in an FHE context, publish them to the fhe-hq Files page so they appear automatically.

**Gate (all must hold):**
- FHE context — the worktree is under `~/Projects/FirstHealth/`, or `$CANONICAL_NAME` is an FHE repo.
- The artifact is a shareable deliverable you created this session — a report, briefing, deck, dashboard, standalone `.html`/`.md`/`.pdf`/`.json`/image.

**Never push:** project docs (TASKS.md, PROGRESS.md, CONTEXT.md, PLAN.md, BRAIN.md, VISION.md, CLAUDE.md, session shards, handoffs), source code, configs, `.claude/*`, scratch/backup files. When unsure whether something is a shareable deliverable, skip it.

For each deliverable (absolute path):
```bash
bash ~/.claude/skills/_lib/fhe-hq-file-push.sh "<abs-path>" --name "<human title>" --tags "<comma,tags>"
```
The shared engine self-gates (FHE-path + exclusion list), auto-detects `file_type`, uploads to the public `fhe-files` bucket, and upserts the `public.fhe_files` row — idempotent on `storage_path` (re-pushing updates, never duplicates). Exit codes: `0` pushed · `3` skipped by a gate (stay silent) · `4`/`5` key or upload error (surface it).

> The `fhe-files` bucket is PUBLIC-read — anything pushed is fetchable by URL. Push only share-intended artifacts.

### Step 7: Confirm

```
Wrapped up: $CANONICAL_NAME / $BRANCH
- Shard:      .claude/sessions/$SHARD_KEY.md
- Handoff:    .claude/handoffs/$SHARD_KEY.md
- Brain:      <N> captures (project_ref=$CANONICAL_NAME, branch=$BRANCH)
- .gitignore: <unchanged | negations appended | negations appended (no-op) | skipped (generated)>
- Files:      <N pushed to fhe-hq Files — list each link | none>
- PROGRESS.md: untouched — consolidates at merge to main
- Next: /pickup on this branch, or /push to send branch to remote
```

## Rules

- NEVER write to PROGRESS.md from /wrap-up. PROGRESS.md is consolidated only at merge to main, by /push.
- NEVER write to `.claude/handoff.md` (singular path). Use `.claude/handoffs/$SHARD_KEY.md`.
- ALWAYS use `$CANONICAL_NAME` for `project_ref` — never the worktree basename.
- ALWAYS prefix brain `text` with `[branch:... | wt:... | project:...]` so cross-branch context is searchable.
- BRAIN.md entries MUST use `### YYYY-MM-DD / $BRANCH` heading so concurrent branches don't collide.
- If `CANONICAL_ROOT == WORKTREE_ROOT` (no worktree in play), the workflow degrades cleanly to single-branch behavior — same outputs, no special cases.
- NEVER skip brain captures — they compound across projects.
- ALWAYS write the handoff file — this is what makes /pickup work.
- If TASKS.md doesn't exist, skip task updates (don't create one just for this).
- Keep the handoff kickstart prompt SPECIFIC — file paths, function names, line numbers.
- Auto-push deliverables to fhe-hq Files only in FHE context; the engine (`_lib/fhe-hq-file-push.sh`) is the single source of truth for gating + upload. Never publish project docs, source, configs, or scratch files. The bucket is public — share-intended only.
